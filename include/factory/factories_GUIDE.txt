----------------------------------------------------------------
FILE: EnemyFactory.h
----------------------------------------------------------------
OWNER        : Student B
DEPENDS ON   : Enemy.h, all concrete enemy headers
               (Botom.h, FlyngFoogaFoog.h, Tornado.h,
               Mogera.h, MogeraChild.h, Gamakichi.h)
NEEDED BY    : LevelManager.h

DESIGN PATTERN: Factory Pattern
    This is one of the required design patterns in the spec.
    It must be documented in your project report.

WHAT IT IS:
Creates any enemy type by name without LevelManager
needing to know the concrete class directly.
This is the Factory design pattern.

WHY THIS EXISTS — THE PROBLEM IT SOLVES:
Without EnemyFactory, LevelManager would need this:

    if (type == "botom")        return new Botom();
    else if (type == "flyng")   return new FlyngFoogaFoog();
    else if (type == "tornado") return new Tornado();
    else if (type == "mogera")  return new Mogera();
    ...

This is bad design for three reasons:
    1. Adding a new enemy means changing LevelManager
    2. LevelManager knows too much about concrete classes
    3. Violates the Open/Closed Principle

With EnemyFactory:
    Enemy* e = EnemyFactory::create(type);

LevelManager never changes when new enemies are added.
Only EnemyFactory needs a new registration line.

WHAT IT HOLDS:
    map<string, CreatorFunction> registry
    — maps string names to functions that create enemies

    Example registry contents:
        "botom"   → function that returns new Botom()
        "flyng"   → function that returns new FlyngFoogaFoog()
        "tornado" → function that returns new Tornado()
        "mogera"  → function that returns new Mogera()
        "child"   → function that returns new MogeraChild()
        "gama"    → function that returns new Gamakichi()

KEY METHODS:

    create(type: string) : Enemy*
        — looks up type in registry
        — calls the creator function
        — returns the new enemy object
        — returns nullptr if type string is unknown
        — LevelManager calls this for every enemy spawn

    registerType(name, creatorFunction)
        — adds a new enemy type to the registry
        — called once at game startup for each type
        — Student A calls this in Game constructor

HOW REGISTRATION WORKS IN PRACTICE:
In Game.cpp constructor or init function
(Student A writes this part):

    EnemyFactory::registerType("botom",
        []() -> Enemy* { return new Botom(); });

    EnemyFactory::registerType("flyng",
        []() -> Enemy* { return new FlyngFoogaFoog(); });

    EnemyFactory::registerType("tornado",
        []() -> Enemy* { return new Tornado(); });

    EnemyFactory::registerType("mogera",
        []() -> Enemy* { return new Mogera(); });

    EnemyFactory::registerType("child",
        []() -> Enemy* { return new MogeraChild(); });

    EnemyFactory::registerType("gama",
        []() -> Enemy* { return new Gamakichi(); });

Then LevelManager spawns enemies like this:

    Enemy* e = EnemyFactory::create("botom");

HOW LEVELMANAGER USES THE FACTORY:
LevelManager reads enemy type strings from level data.
It never uses new Botom() or new Tornado() directly.
Everything goes through EnemyFactory::create().

    for (int i = 0; i < levelConfig.enemyCount; i++) {
        string type = getEnemyTypeForLevel(level, i);
        Enemy* e = EnemyFactory::create(type);
        if (e != nullptr) enemies[i] = e;
    }

WHAT IT DELIVERS AFTER IMPLEMENTATION:
LevelManager can spawn any enemy from a string name.
Adding a new enemy type in the future requires:
    1. Create the new class (inherits Enemy)
    2. Register it in EnemyFactory
    3. Nothing else changes anywhere

This satisfies the spec requirement:
"Design scalable code that can easily grow from
10 levels to 20 or more without significant refactoring."

SIMPLE EXPLANATION:
EnemyFactory is like a restaurant menu. You say
"I want item number 3" (the type name). The kitchen
(factory) makes it and brings it to you. You never
need to know how it is cooked (constructed).

SINGLETON OR STATIC:
EnemyFactory can be implemented as either:
    Option A — Static class (all methods static, no object needed)
    Option B — Singleton (one global instance)

Recommended: Static class. Simpler to use.
    EnemyFactory::create("botom");  // no object needed

RULES:
- create() must return nullptr for unknown type strings.
  Never crash on unknown input — log a warning instead.
- All enemy types must be registered BEFORE
  LevelManager.loadLevel() is called for the first time.
  Do all registration in Game's constructor or init().
- Student B writes the factory mechanism.
  Student A registers the concrete enemy types
  since Student A owns those classes.
- The string names used in registration must match
  exactly what LevelManager passes to create().
  Agree on the exact strings with Student A.
