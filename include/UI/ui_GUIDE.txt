CRITICAL RULE FOR ALL SCREEN FILES:
Never put game logic inside a screen.
Screens only display things and pass input to the right place.
All actual logic lives in Game, LevelManager, Player,
and the systems. If you find yourself writing collision
detection or score calculation inside a screen — stop
and move it to the correct class.

FILE 1: Screen.h
----------------------------------------------------------------
OWNER        : Student B
DEPENDS ON   : SFML
NEEDED BY    : All 8 screen subclasses, Game.h

WHAT IT IS:
Abstract base class for every screen in the game.
Game.h holds a pointer to the currently active Screen
and calls update(), draw(), and handleInput() on it
every frame without knowing which specific screen it is.

PURE VIRTUAL METHODS:
    update(float dt)         — logic every frame
    draw()                   — draw this screen
    handleInput(sf::Event&)  — process one input event

TRANSITION SYSTEM:
When a screen wants to switch to another screen it sets:
    transitioning = true
    nextScreen = "GameScreen"  (or whatever screen name)

Game.cpp reads isTransitioning() every frame and
swaps the active screen when true.

PROTECTED MEMBERS:
    sf::RenderWindow& window
    sf::Font font
    bool        transitioning = false
    string      nextScreen    = ""

KEY METHODS:
    isTransitioning() : bool
    getNextScreen() : string

WHAT IT DELIVERS AFTER IMPLEMENTATION:
Game.cpp can hold one Screen pointer and call the
same three methods regardless of which screen is active.
This is polymorphism working in the UI layer.

FILE 2: GameUI.h (previously called HUD)
----------------------------------------------------------------
OWNER        : Student B
DEPENDS ON   : SFML, ScoreSystem.h, GemSystem.h, LevelManager.h
NEEDED BY    : GameScreen.h

WHAT IT IS:
The overlay drawn on top of the gameplay every frame.
Shows all live game information without interrupting play.
This is not a Screen subclass — it is drawn by GameScreen
on top of the game world.

WHAT IT DISPLAYS:
    Score          — top left, reads from ScoreSystem every frame
    Lives          — top left below score, drawn as icons
    Gem count      — top right, reads from GemSystem every frame
    Level          — top center, e.g. "Level 3 / 10"
    Power-up info  — bottom, active power-up name and timer
    Boss health bar— top center, ONLY on levels 5 and 10

KEY METHODS:
    update(dt)           — refreshes all displayed values
    draw(window)         — draws all elements on screen

BOSS HEALTH BAR RULE:
Every frame GameUI checks LevelManager.isBossLevel().
If true draw the health bar.
If false do not draw it.
Never hardcode level numbers 5 and 10 inside GameUI.
Always ask LevelManager.

FONT ASSIGNMENTS:
    Score numbers     → PressStart2P
    Lives count       → PressStart2P
    Gem count         → PressStart2P
    Level text        → Silkscreen
    Power-up name     → VT323
    Boss health label → Silkscreen

WHAT IT DELIVERS AFTER IMPLEMENTATION:
Player always sees their current score, lives, gems,
and level. Boss health bar appears automatically
on boss levels. Power-up timer counts down visibly.

FILE 5: GameScreen.h
----------------------------------------------------------------
OWNER        : Student B
DEPENDS ON   : Screen.h, GameUI.h, LevelManager.h
NEEDED BY    : Game.h (main gameplay screen)

WHAT IT IS:
The wrapper screen during actual gameplay.
Does NOT contain game logic. Just coordinates
the game systems during play.

WHAT IT DOES EVERY FRAME:
    1. Calls game.update(dt) to update all game objects
    2. Calls game.draw() to draw the game world
    3. Calls gameUI.draw() to overlay the HUD on top
    4. Checks if level is cleared → transition to LevelCompleteScreen
    5. Checks if game over → transition to GameOverScreen

KEY INPUT HANDLING:
    Escape key → transitions to "PauseMenuScreen"
    F1 key     → toggles HitBox debug mode globally

WHAT IT DELIVERS AFTER IMPLEMENTATION:
The game runs inside this screen. HUD overlays correctly.
Pause works. Level complete and game over transitions work.

FILE 6: PauseMenuScreen.h
----------------------------------------------------------------
OWNER        : Student B
DEPENDS ON   : Screen.h, ShopSystem.h, AuthManager.h,
               ProgressRepository.h
NEEDED BY    : GameScreen.h (pause transition target)

WHAT IT IS:
Appears when Escape is pressed during gameplay.
Game world freezes underneath — nothing updates
while paused. Returns to GameScreen on resume.

BUTTONS AND ACTIONS:
    Resume         → transitions back to "GameScreen"
    Open Shop      → transitions to "ShopScreen"
    Save Game      → calls ProgressRepository.saveProgress()
                     shows "Saved!" confirmation briefly
    Logout         → calls AuthManager.logout()
                     transitions to "MainMenuScreen"
    Exit to Menu   → transitions to "MainMenuScreen"
                     without saving (warn the player first)

WHAT IT DELIVERS AFTER IMPLEMENTATION:
Player can pause, shop, save, and logout cleanly
from any point in the game without losing progress.
