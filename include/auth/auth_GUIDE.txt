This folder handles everything about who is playing.
Login, registration, session tracking, and logout
all live here. Two files work together.

CRITICAL SECURITY RULE:
NEVER store plain text passwords anywhere.
Always hash before storing. Always hash before comparing.
If the professor checks your users.txt and sees plain
text passwords that is an automatic deduction.

Write order: Session.h → AuthManager.h

----------------------------------------------------------------
FILE 1: Session.h
----------------------------------------------------------------
OWNER        : Student B
DEPENDS ON   : Nothing
NEEDED BY    : AuthManager.h, Game.h, all screens that
               need to know who is logged in,
               ProgressRepository.h

WHAT IT IS:
A simple data container. Holds information about
whoever is currently logged in. No file reading,
no logic, no SFML. Just data and simple getters.

AuthManager creates and clears Session objects.
Everything else only reads from Session — never writes.

WHAT IT HOLDS:
    string username    — logged in player's display name
    int    userId      — unique ID from users.txt
    bool   loggedIn    — is anyone currently playing?

KEY METHODS:
    isLoggedIn() : bool      — is someone logged in?
    getUsername() : string   — who is logged in?
    getUserId() : int        — their database ID
    clear()                  — called on logout, resets everything

HOW clear() WORKS:
    username  = ""
    userId    = -1
    loggedIn  = false

WHAT IT DELIVERS AFTER IMPLEMENTATION:
Any class that needs to know who is playing asks Session.
No class needs to touch AuthManager or the database
just to find the current player's name.

SIMPLE EXPLANATION:
Session is like a name badge you wear at a conference.
While you are wearing it anyone can read your name.
When you leave (logout) the badge is taken away
and reset for the next person.

USAGE EXAMPLE:
    if (session.isLoggedIn()) {
        string name = session.getUsername();
        // show welcome message with name
    }

RULES:
- Session holds data only. No logic, no file I/O.
- Only AuthManager creates Session objects and
  calls clear(). No other class writes to Session.
- All other classes only READ from Session via getters.
- A single global Session object lives in Game.
  AuthManager has a pointer to it.

----------------------------------------------------------------
FILE 2: AuthManager.h
----------------------------------------------------------------
OWNER        : Student B
DEPENDS ON   : Session.h, UserRepository.h
NEEDED BY    : MainMenuScreen.h, PauseMenuScreen.h, Game.h

WHAT IT IS:
Handles all authentication logic. Login, registration,
and logout. Never touches files directly — always
goes through UserRepository.

WHAT IT HOLDS:
    Session* session         — pointer to the active session
                               (points to Game's Session object)

KEY METHODS:

    login(username, password) : bool
        — validates that username and password are not empty
        — calls UserRepository.findByUsername(username)
        — if user not found → return false
        — hashes the entered password
        — compares hash with stored hash in user record
        — if match → fills Session with user data, return true
        — if no match → return false

    registerUser(username, password) : bool
        — validates username and password not empty
        — checks minimum password length (suggest: 6 chars)
        — calls UserRepository.usernameExists(username)
        — if username taken → return false
        — hashes the password
        — calls UserRepository.create(username, hash, "")
        — return true on success

    logout()
        — saves current progress before logging out
          (calls ProgressRepository.saveProgress())
        — calls session->clear()

    isLoggedIn() : bool
        — shortcut: return session->isLoggedIn()

    getSession() : Session*
        — returns pointer to current session

    hashPassword(password) : string    ← PRIVATE METHOD

PASSWORD HASHING ALGORITHM (djb2):
Use this exact algorithm. Simple but sufficient for
a student project:

    string AuthManager::hashPassword(const string& pass) {
        unsigned long hash = 5381;
        for (char c : pass) {
            hash = ((hash << 5) + hash) + c;
        }
        return to_string(hash);
    }

This converts the password to a number string.
Store that number string in users.txt.
When logging in hash the entered password the same
way and compare the resulting strings.

EXAMPLE LOGIN FLOW:
    User enters: username="zain", password="mypass123"
    
    1. UserRepository.findByUsername("zain") → finds record
    2. Record contains: passwordHash="12345678901"
    3. hashPassword("mypass123") → "12345678901"
    4. "12345678901" == "12345678901" → match → login success
    5. session.username = "zain"
    6. session.userId = 1
    7. session.loggedIn = true
    8. return true

EXAMPLE FAILED LOGIN FLOW:
    User enters: username="zain", password="wrongpass"
    
    1. UserRepository.findByUsername("zain") → finds record
    2. hashPassword("wrongpass") → "99887766554"
    3. "99887766554" != "12345678901" → no match
    4. return false
    5. MainMenuScreen shows "Incorrect password"

ERROR MESSAGES TO RETURN OR SIGNAL:
    "User not found"        — username does not exist
    "Incorrect password"    — username found but hash mismatch
    "Username taken"        — during registration
    "Password too short"    — less than 6 characters
    "Fields cannot be empty"— empty username or password

Consider returning an enum or string code instead of
just bool so the screen can show the right message.

    enum class AuthResult {
        SUCCESS,
        USER_NOT_FOUND,
        WRONG_PASSWORD,
        USERNAME_TAKEN,
        INVALID_INPUT
    };

WHAT IT DELIVERS AFTER IMPLEMENTATION:
Players can create accounts and log in securely.
Passwords are never stored in plain text.
Session tracks the current player for the whole game.
Logout saves progress and clears the session cleanly.

SIMPLE EXPLANATION:
AuthManager is like a hotel front desk. You give
your name and password. If correct you get a room
key (Session). When you check out (logout) the key
is deactivated and your bill is settled (progress saved).

RULES:
- NEVER store plain text passwords. Always use hashPassword().
- Always validate inputs before touching the database.
  Empty string checks save you from pointless file reads.
- AuthManager never reads files directly.
  Always calls UserRepository methods.
- Save progress in logout() before clearing Session.
  If you clear Session first you lose the userId needed to save.
- AuthManager does not know about screens. It just
  returns success/failure. The screen decides what to show.
