This folder contains the two base classes that every
single game object inherits from directly or indirectly.
Player, Enemy, Projectile, PowerUp — they all start here.

Write order: HitBox.h → Entity.h

----------------------------------------------------------------
FILE 1: HitBox.h
----------------------------------------------------------------
OWNER        : Student A
DEPENDS ON   : SFML (sf::FloatRect, sf::RenderWindow)
NEEDED BY    : Entity.h, CollisionDetector.h, every entity

WHAT IT IS:
A rectangle that wraps around every game object.
Used for collision detection instead of the visible sprite.
This rectangle is invisible during normal gameplay but
can be shown in debug mode by pressing F1.

WHAT IT HOLDS:
    sf::FloatRect rect       — the actual collision rectangle
    bool visible             — is debug mode on?
    sf::Color debugColor     — color when drawn in debug mode
    static bool debugVisible — global toggle for ALL hitboxes

KEY METHODS:
    intersects(other)   — returns true if two hitboxes overlap
    draw(window)        — draws the colored outline in debug mode
    toggle()            — flips debug visibility
    setRect(r)          — updates the rectangle position/size
    getRect()           — returns the rectangle

WHY IT IS SEPARATE FROM THE SPRITE:
The spec specifically says the hitbox must be separate
from the visual sprite. The visual can be larger than
the collision area. This makes the game feel fair because
players do not get hit by pixels that visually miss them.

    Example:
    Botom sprite  = 64x64 pixels (what you see)
    Botom hitbox  = 48x48 pixels (what actually collides)

DEBUG COLORS PER ENTITY TYPE:
    Green  = player hitboxes
    Red    = enemy hitboxes
    Yellow = projectile hitboxes
    Blue   = platform hitboxes

WHAT IT DELIVERS AFTER IMPLEMENTATION:
Every entity in the game automatically has a collision
rectangle. Pressing F1 shows all rectangles at once
so you can debug collisions visually.

SIMPLE EXPLANATION:
Think of HitBox like an invisible force field around
each character. When two force fields touch something
happens. F1 makes all force fields visible.

IMPORTANT RULE ABOUT debugVisible:
It is a static bool — meaning ONE value shared by ALL
HitBox objects at the same time. When F1 is pressed
and you flip this one bool every single hitbox in the
game shows or hides instantly. You never loop through
entities to toggle them one by one.

RULES:
- debugVisible must be static.
- Never put collision logic inside HitBox itself.
  HitBox only stores the rectangle and draws it.
  CollisionDetector does the actual logic.
- Always update the hitbox position when the entity moves.
  A hitbox that does not follow its entity is useless.

FILE 2: Entity.h
----------------------------------------------------------------
OWNER        : Student A
DEPENDS ON   : HitBox.h, SFML
NEEDED BY    : Player.h, Enemy.h, Projectile.h, PowerUp.h

WHAT IT IS:
The base class for every single object in the game.
If it exists on screen and has a position it inherits
from Entity. It guarantees every game object has the
same basic properties.

WHAT IT HOLDS:
    sf::Vector2f position  — where the entity is on screen
    sf::Vector2f velocity  — how fast and which direction
    HitBox hitbox          — its collision rectangle
    bool active            — is it still in the game?

KEY METHODS (pure virtual — must be overridden):
    update(float dt)              — logic every frame
    draw(sf::RenderWindow& w)     — drawing every frame

OTHER METHODS:
    getPosition()    — returns current position
    getHitBox()      — returns the hitbox rectangle
    isActive()       — returns whether entity is still alive
    setActive(bool)  — activates or deactivates entity

WHAT IT DELIVERS AFTER IMPLEMENTATION:
A guaranteed common interface for every game object.
The game loop can call update() and draw() on any entity
without knowing exactly what type it is. This is
polymorphism working in practice.

SIMPLE EXPLANATION:
Think of Entity like a job description that says
"every employee must be able to do these two things:
update yourself and draw yourself."
Every specific class fills in exactly how they do it.

HIERARCHY:
    Entity (abstract base — this file)
        ├── Player
        ├── Enemy (abstract)
        │       ├── Botom
        │       ├── FlyingEnemy (abstract)
        │       │       ├── FlyngFoogaFoog
        │       │       └── Tornado
        │       └── Boss (abstract)
        │               ├── Mogera
        │               └── Gamakichi
        ├── Projectile (abstract)
        │       ├── Snowball
        │       ├── Knife
        │       └── ArtilleryRocket
        └── PowerUp (abstract)
                ├── SpeedBoost
                ├── SnowballPower
                ├── DistanceIncrease
                ├── BalloonMode
                └── ExtraLife

RULES:
- Entity is abstract. You can never create an Entity object directly.
  You can only create objects of its concrete child classes.
- update() and draw() must be pure virtual (= 0).
  This forces every child class to implement them.
- Always call hitbox.setRect() whenever position changes
  so the hitbox follows the entity.
- The active flag is how you remove entities from the game.
  Never delete them mid-loop. Just set active = false
  and clean them up at the end of the frame.
