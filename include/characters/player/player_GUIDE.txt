----------------------------------------------------------------
FILE 1: StatsComponent.h
----------------------------------------------------------------
OWNER        : Student A
DEPENDS ON   : Nothing (no SFML needed here)
NEEDED BY    : Player.h, all PowerUp classes

WHAT IT IS:
A simple data container that holds all upgradeable
stats for one player. It has no drawing, no input,
no SFML. Just numbers and flags that power-ups can change.

WHAT IT HOLDS:
    float speed              — movement speed (default: 200.0f)
    int   snowballPower      — hits to fully encase enemy (default: 2)
    float snowballDistance   — how far snowball travels (default: medium)
    bool  balloonMode        — is player floating? (default: false)

KEY METHODS:
    applyPowerUp(PowerUp*)   — applies a power-up's effect
    resetTemporary()         — removes level-scoped effects at level end
    getSpeed()               — returns current speed
    getSnowballPower()       — returns current power
    getSnowballDistance()    — returns current distance

WHY IT IS SEPARATE FROM PLAYER:
If all stats were inside Player, power-ups would need
direct access to Player's private variables.
By keeping stats in their own class, power-ups only
talk to StatsComponent and never touch Player directly.
This keeps things clean and separated.

WHAT IT DELIVERS AFTER IMPLEMENTATION:
Power-ups can be applied and removed cleanly without
touching Player's movement or drawing code.
Stats always reset correctly at level end.

SIMPLE EXPLANATION:
Think of StatsComponent like a player's character sheet
in a board game. It holds all the numbers. Power-ups
change the numbers. The player reads the numbers to
decide how fast to move or how far to throw.

POWER-UP EFFECTS ON STATS:
    SpeedBoost      → speed *= 1.5f
    SnowballPower   → snowballPower = 1
    DistanceIncrease→ snowballDistance = SCREEN_WIDTH
    BalloonMode     → balloonMode = true
    ExtraLife       → does not touch stats (goes to Player directly)

TEMPORARY vs PERMANENT effects:
    Temporary (reset at level end):
        snowballPower, snowballDistance, balloonMode
    Permanent (never reset):
        speed boost timer only, base speed unchanged
        extra lives go to Player not StatsComponent

RULES:
- Never let any class other than PowerUp subclasses
  and Player directly modify these values.
- resetTemporary() is called by LevelManager at level end.
  Never call it manually from anywhere else.

FILE 2: SnowballLauncher.h
----------------------------------------------------------------
OWNER        : Student A
DEPENDS ON   : StatsComponent.h, Snowball.h, SFML
NEEDED BY    : Player.h

WHAT IT IS:
Handles everything about throwing and managing snowballs.
Player owns one SnowballLauncher and calls fire() when
the throw button is pressed.

WHAT IT HOLDS:
    StatsComponent* stats    — reference to player's stats
    Snowball snowballs[]     — array of active snowballs in flight
    int activeCount          — how many snowballs are currently flying

KEY METHODS:
    fire(direction)          — creates a new snowball going left or right
    update(dt)               — updates all active snowballs every frame
    draw(window)             — draws all active snowballs
    getSnowballs()           — returns array for collision detection
    setPower(p)              — updates power from StatsComponent
    setDistance(d)           — updates distance from StatsComponent

SNOWBALL WRAPPING BEHAVIOR:
When a snowball reaches the right edge of the screen
it reappears from the left edge and continues moving.
Same for left to right. This is the classic Snow Bros
mechanic and SnowballLauncher handles this wrapping.

WHAT IT DELIVERS AFTER IMPLEMENTATION:
Player can throw snowballs that travel across the screen,
wrap around edges, and interact with enemies.
Multiple snowballs can be in flight at the same time.

SIMPLE EXPLANATION:
Think of SnowballLauncher like a magazine in a gun.
Player pulls the trigger (fire()) and the launcher
manages all the bullets (snowballs) that are currently
in the air.

RULES:
- Maximum snowballs in flight at once should be capped
  to a reasonable number (suggest: 3).
- Always read distance and power from StatsComponent
  not from hardcoded values. This way power-ups
  automatically affect throwing behavior.
- Snowballs that have expired or hit something must
  be marked inactive and reused for the next throw.
  Never create new Snowball objects every throw.

FILE 3: Player.h
----------------------------------------------------------------
OWNER        : Student A
DEPENDS ON   : Entity.h, StatsComponent.h, SnowballLauncher.h,
               InputManager.h, CollisionDetector.h, SFML
NEEDED BY    : Game.h, LevelManager.h, all collision checks

WHAT IT IS:
The playable character. The most visible class in
the game from the player's perspective. Handles
movement, jumping, gravity, drawing the sprite,
and managing lives.

WHAT IT HOLDS:
    StatsComponent stats         — upgradeable stats
    SnowballLauncher launcher    — handles snowball throwing
    int lives                    — current remaining lives
    bool facingRight             — which way player faces
    bool onGround                — is player standing on something?
    int playerId                 — 0 for Player 1, 1 for Player 2
    sf::Sprite sprite            — the visible character sprite
    sf::Texture texture          — sprite texture loaded from file

KEY METHODS:
    update(dt)          — reads input, applies physics, updates position
    draw(window)        — draws sprite at current position
    jump()              — applies upward velocity if on ground
    throwSnowball()     — calls launcher.fire()
    takeDamage()        — removes one life, fires PLAYER_HIT event
    getLives()          — returns current life count
    addLife(n)          — adds n lives (used by ExtraLife power-up)
    reset()             — resets position and state for new level

PHYSICS BEHAVIOR:
    Gravity:
        Every frame add downward velocity.
        velocity.y += GRAVITY * dt
        position.y += velocity.y * dt
    
    Jump:
        On jump key press set velocity.y = -JUMP_FORCE
        Only allowed when onGround is true.
    
    Landing:
        CollisionDetector checks platforms every frame.
        When on platform set velocity.y = 0 and onGround = true.
    
    Screen wrap:
        If position.x > SCREEN_WIDTH  → position.x = 0
        If position.x < 0             → position.x = SCREEN_WIDTH

SPRITE BEHAVIOR:
    When facingRight is true  → draw sprite normally
    When facingRight is false → flip sprite horizontally
    Use sf::Sprite setScale(-1, 1) to flip

TWO PLAYER SUPPORT:
    Player 1 (playerId = 0) → reads Player 1 keys from InputManager
    Player 2 (playerId = 1) → reads Player 2 keys from InputManager
    Both are completely separate Player objects.
    They do not share stats, lives, or launchers.

WHAT IT DELIVERS AFTER IMPLEMENTATION:
A fully controllable character that moves, jumps,
falls with gravity, lands on platforms, wraps around
the screen, throws snowballs, and loses lives when hit.

SIMPLE EXPLANATION:
Player is the character you control. Everything the
player does on screen — moving, jumping, throwing —
is handled by this class.

HIERARCHY:
    Entity
        └── Player
                ├── has StatsComponent
                ├── has SnowballLauncher
                └── uses InputManager

RULES:
- Player never does collision detection itself.
  It asks CollisionDetector and responds to the result.
- Player never updates the score. It fires PLAYER_HIT
  event via EventBus and ScoreSystem listens.
- Player 1 and Player 2 are separate objects.
  Never share any state between them.
- Always check onGround before allowing a jump.
  Otherwise player can jump in mid-air.
