FILE 1: Enemy.h
----------------------------------------------------------------
OWNER        : Student B (Minahil) — writes the abstract base
               Student A (Zain)    — writes all concrete subclasses
DEPENDS ON   : Entity.h, SFML
NEEDED BY    : Every enemy class, CollisionDetector, LevelManager,
               ScoreSystem, EnemyFactory

WHAT IT IS:
The abstract base class for every enemy in the game.
Student B defines what every enemy MUST be able to do.
Student A implements HOW each specific enemy does it.

PURE VIRTUAL METHODS (Student B defines, Student A implements):
    update(float dt)         — AI logic every frame
    draw(RenderWindow& w)    — draw enemy sprite
    takeDamage(int amt)      — receive a hit
    isDefeated() : bool      — has this enemy been killed?
    getScoreValue() : int    — how many points for killing this?
    getHitBox() : FloatRect  — collision rectangle

SHARED METHODS (Student B implements in Enemy.cpp):
    isEncased() : bool       — is enemy fully covered in snow?
    applySnow()              — add one layer of snow cover

PROTECTED MEMBERS (both students use these):
    sf::Vector2f position
    sf::Vector2f velocity
    int   health    = 1
    float speed     = 100.f
    bool  defeated  = false
    bool  encased   = false
    int   snowCover = 0    (0=none, 1=partial, 2=fully encased)

WHAT IT DELIVERS AFTER IMPLEMENTATION:
A guaranteed interface that lets the game loop call
update() and draw() on any enemy without knowing
its specific type. CollisionDetector can check any
enemy's hitbox the same way. ScoreSystem can get
any enemy's score value the same way.

SIMPLE EXPLANATION:
Enemy.h is a job description. It says every enemy
must be able to do these specific things. Each
concrete class fills in the details.

RULES FOR STUDENT B:
- Only write the abstract base here. No AI logic.
- Make sure all methods listed above are pure virtual (= 0).
- snowCover values: 0 = not hit, 1 = partial, 2 = fully encased.
- Never change method signatures after Student A starts coding.

RULES FOR STUDENT A:
- Every concrete enemy class must implement ALL pure virtual methods.
- Never add collision logic inside enemy classes.
  Always use CollisionDetector.
- Always fire ENEMY_KILLED event via EventBus on defeat.
  Never update score directly.

FILE 2: Botom.h
----------------------------------------------------------------
OWNER        : Student A
DEPENDS ON   : Enemy.h
NEEDED BY    : EnemyFactory.h, LevelManager.h

WHAT IT IS:
The most basic enemy. Walks left and right on platforms.
Falls with gravity. Turns at walls and edges.
Everything else inherits from or builds on Botom's behavior.

ADDITIONAL MEMBERS:
    int   dirX       — current movement direction (-1 left, 1 right)
    float walkTimer  — timer for random direction changes

KEY BEHAVIOR:
    Random walk:
        Moves in one direction until it hits a wall,
        falls off an edge, or the walkTimer expires.
        Then picks a new direction randomly.
        Must feel organic — not robotic.
    
    Gravity:
        Falls when not on a platform.
        Same gravity as Player.
    
    Encasing:
        Takes 2 snowball hits by default to fully encase.
        (1 hit if player has SnowballPower active)
        snowCover goes 0 → 1 → 2 on each hit.

SCORE RANGE: 100 to 500 points (random in this range on kill)

COLOR VARIANTS (levels 6-9):
    Red    (base)   levels 1-5  : speed 1.0x, 2 hits to encase
    Green           levels 6-7  : speed 1.25x, 3 hits to encase
    Blue            levels 7-8  : speed 1.5x, 4 hits to encase
    Purple          levels 8-9  : speed 1.75x, 5 hits to encase

WHAT IT DELIVERS AFTER IMPLEMENTATION:
A moving enemy that walks around, reacts to snowballs,
can be encased, and drops score when killed.
This is the minimum viable enemy for the game to be playable.

SIMPLE EXPLANATION:
Botom is like a Roomba — it bumps into walls and
changes direction. Simple but forms the foundation
for all other enemies.

RULES:
- Master Botom before touching any other enemy.
  Everything else builds on this.
- The random walk must feel natural. Use randomized
  timers not perfectly regular intervals.
- Always drop a random power-up or gem on defeat.
  Fire GEM_DROPPED event via EventBus.

FILE 3: FlyingEnemy.h
----------------------------------------------------------------
OWNER        : Student A
DEPENDS ON   : Enemy.h
NEEDED BY    : FlyngFoogaFoog.h, Tornado.h

WHAT IT IS:
An abstract middle layer between Enemy and the two
flying enemy types. It adds the flight state machine
so FlyngFoogaFoog and Tornado do not repeat this code.

ADDITIONAL MEMBERS:
    bool  flying         — is enemy currently in flight?
    float flightTimer    — how long until next state switch
    float flightDuration — how long one flight lasts

KEY METHODS (virtual — can be overridden):
    enterFlight()    — switches to flight mode
    exitFlight()     — returns to ground mode
    isFlying() : bool

FLIGHT STATE MACHINE:
    GROUND ──(timer expires)──► FLIGHT ──(duration expires)──► GROUND
                                    ↑                               |
                                    └───────────────────────────────┘

WHAT IT DELIVERS AFTER IMPLEMENTATION:
A reusable flight system that both FlyngFoogaFoog
and Tornado inherit automatically without duplicating code.

SIMPLE EXPLANATION:
FlyingEnemy is like a template for any enemy that
can take off and land. It handles the timing of
when to fly and when to land so each subclass
only needs to define what happens during flight.

RULES:
- This class is still abstract. update() is still pure virtual.
  FlyngFoogaFoog and Tornado implement it.
- Flight timers should be slightly randomized so all
  flying enemies do not take off at the same time.

FILE 4: FlyngFoogaFoog.h
----------------------------------------------------------------
OWNER        : Student A
DEPENDS ON   : FlyingEnemy.h
NEEDED BY    : EnemyFactory.h, LevelManager.h

WHAT IT IS:
Inherits Botom's ground behavior and FlyingEnemy's
flight system. Adds free 8-direction movement in flight.
Harder to hit because its flight path is unpredictable.

ADDITIONAL MEMBERS:
    float groundTimer    — time spent on ground before taking off
    sf::Vector2f flightDir — current flight direction

KEY BEHAVIOR:
    On ground  : same as Botom (random walk)
    In flight  : moves freely in 8 directions across entire screen
    Landing    : returns to nearest platform when flight ends

SCORE RANGE: 200 to 800 points (random in this range on kill)

WHAT IT DELIVERS AFTER IMPLEMENTATION:
An enemy that alternates between walking on platforms
and flying freely making it challenging to predict and hit.

SIMPLE EXPLANATION:
Think of FlyngFoogaFoog like a bird that walks on
the ground most of the time but occasionally takes
off and flies unpredictably before landing again.

FILE 5: Tornado.h
----------------------------------------------------------------
OWNER        : Student A
DEPENDS ON   : FlyingEnemy.h, Knife.h
NEEDED BY    : EnemyFactory.h, LevelManager.h

WHAT IT IS:
Inherits everything from FlyngFoogaFoog and adds
two dangerous new behaviors — randomized teleport
speed and knife throwing.

ADDITIONAL MEMBERS:
    float knifeTimer      — time until next knife throw
    float teleportSpeed   — randomized each flight jump
    Knife* knife          — current active knife projectile

KEY BEHAVIOR:
    Teleport flights:
        Flight jumps happen at randomized speeds.
        Sometimes very slow, sometimes nearly instant.
        Position becomes very unpredictable.
    
    Knife throwing:
        At set intervals fires a Knife toward the
        player's last known position.
        Knife travels in a straight line.
        Hitting player costs one life.
        Fires PLAYER_HIT event via EventBus.

SCORE RANGE: 300 to 1200 points (random in this range on kill)

WHAT IT DELIVERS AFTER IMPLEMENTATION:
The most dangerous standard enemy. Combines unpredictable
movement with a long-range attack. Forces player to
both dodge movement and watch for incoming knives.

SIMPLE EXPLANATION:
Tornado is like FlyngFoogaFoog but angrier. It moves
unpredictably AND throws knives at you.

RULES:
- Knife direction is calculated ONCE at throw time
  using player's position at that exact moment.
  The knife does not home in — it travels in a
  straight line after that.
- Teleport speed randomized each jump: between 50 and 500.

               Student A (Zain)    — writes Mogera and Gamakichi
DEPENDS ON   : Enemy.h
NEEDED BY    : Mogera.h, Gamakichi.h

Abstract base class for both boss enemies.

ADDITIONAL MEMBERS:
    int maxHealth  = 20
    int phase      = 1
    int gemReward  = 200

    drawHealthBar(w)       — default impl provided by Student B
    getPhase() : int
    getGemReward() : int

HEALTH BAR:
Student B provides a default drawHealthBar() that draws
a rectangle at the top of the screen showing current health.
Mogera and Gamakichi can override this if needed.

WHAT IT DELIVERS AFTER IMPLEMENTATION:
A foundation for boss enemies with health phase support
and an automatic health bar display.

RULES FOR STUDENT B:
- Provide a working default drawHealthBar() implementation.
- onPhaseChange() must be pure virtual.

RULES FOR STUDENT A:
- Never show boss health bar on non-boss levels.
  GameUI checks LevelManager.isBossLevel() first.

FILE 7: MogeraChild.h
----------------------------------------------------------------
OWNER        : Student A
DEPENDS ON   : Enemy.h
NEEDED BY    : Mogera.h

WHAT IT IS:
Small enemy spawned by Mogera during the level 5 fight.
Moves in one fixed direction until it hits a wall or player.
Very simple — no AI complexity.

ADDITIONAL MEMBERS:
    int dirX    — fixed direction set at spawn time (-1 or 1)

KEY BEHAVIOR:
    Spawned by Mogera at its position.
    Moves left or right in a straight line.
    Stops (deactivates) when hitting a wall or player.
    Hitting player fires PLAYER_HIT event.

WHAT IT DELIVERS AFTER IMPLEMENTATION:
Mogera can now actively threaten the player during
the boss fight by spawning small moving hazards.

SIMPLE EXPLANATION:
MogeraChild is like a bowling ball rolling in one
direction. It just goes straight until it hits something.

FILE 8: Mogera.h
----------------------------------------------------------------
OWNER        : Student A
DEPENDS ON   : Boss.h, MogeraChild.h
NEEDED BY    : EnemyFactory.h, LevelManager.h

WHAT IT IS:
The level 5 mid-game boss. Stationary or limited movement.
Primary attack is spawning MogeraChild enemies.
Has a health bar and drops 200 gems on defeat.

ADDITIONAL MEMBERS:
    float spawnTimer      — time until next child spawn
    float spawnInterval   — how often children spawn
    MogeraChild children[]— array of active children

KEY BEHAVIOR:
    Phase 1 (100-50% health):
        Spawns children every 3 seconds.
    Phase 2 (below 50% health):
        Spawns children every 1.5 seconds (faster).
        onPhaseChange() triggers the speed increase.
    
    On defeat:
        Fires BOSS_DEFEATED event with 200 gem reward.
        Background and music return to normal.

SCORE: 5000 points (fixed) + 200 gems

WHAT IT DELIVERS AFTER IMPLEMENTATION:
A complete mid-game boss fight that challenges the player
to avoid spawned children while attacking the boss.

FILE 9: Gamakichi.h
----------------------------------------------------------------
OWNER        : Student A
DEPENDS ON   : Boss.h, ArtilleryRocket.h
NEEDED BY    : EnemyFactory.h, LevelManager.h

WHAT IT IS:
The final boss. Most complex class in the entire project.
Has artillery arms that fire rockets in multiple directions.
Three attack phases that change as health drops.

ADDITIONAL MEMBERS:
    float attackTimer          — time until next attack
    ArtilleryRocket rockets[]  — array of active rockets
    int phaseThresholds[]      — health values that trigger phase changes

KEY BEHAVIOR:
    Phase 1 (100-66% health):
        Fires rockets in 2 directions slowly.
    
    Phase 2 (66-33% health):
        Fires rockets in 4 directions faster.
        onPhaseChange() triggers at 66%.
    
    Phase 3 (below 33% health):
        Fires rockets in 6 directions rapidly.
        onPhaseChange() triggers at 33%.
    
    On defeat:
        Fires BOSS_DEFEATED event with 500 gem reward.
        Game transitions to victory screen.

SCORE: 10000 points (fixed) + 500 gems

WHAT IT DELIVERS AFTER IMPLEMENTATION:
A climactic final boss fight that escalates in
difficulty as the player deals more damage.

RULES:
- Implement phases ONE AT A TIME. Get Phase 1 working,
  then add Phase 2, then Phase 3.
- Test each phase thoroughly before adding the next.
- Rockets must explode on contact — not just disappear.
